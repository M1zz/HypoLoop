For Whom

こんな方のためにつくりました

🧭

教育者・運営者

チームプロジェクト型の教育を運営しているものの、「成果物は出るのに、個人が何を学んだのか分からない」という悩みを抱えている方。運営者個人の力量に頼らず、構造で回る授業を求めている方。

🛠️

チームリード・組織

サイドプロジェクトやハッカソン、オンボーディングで、メンバーが「できることだけやって終わる」役割の固定化を打ち破りたい方。チームの成長と成果物の間の優先順位を明確にしたい方。

📓

勉強会・個人学習者

講義を聞いても何も残らないと感じている方。学ぶ前に仮説を立てて壊すやり方で、一人でも勉強会でも学習の密度を高めたい方。

The Problem

チームは前進しているのに、
個人は学んでいますか?

チームプロジェクトは合理的に回るほど、各自がすでに得意な仕事を担うようになります。成果物の質は上がっても、個人の学習曲線は平らなままです。学びは、問題を解くうちに運よく得られる副産物へと追いやられます。

「価値ある問題を解けば、学びは自然についてくる」
「学びは意図されたときにのみ保証される」

HypoLoopは、この前提の置き換えから出発します。偶発的な学びに賭けていた賭け金を回収し、学習を最初から名前のついた目標にします。

なぜ「問題解決」ではなく「学習」なのか
問題だけを解くと 今の実力で解ける問題だけを解決 → 成果物は積み上がるが実力は横ばい → より難しい問題の前で立ち止まる
学びながら解くと 小さなチャレンジを学びながら解決 → 解決能力そのものが育つ → 究極の難問にまで到達する

問題解決を捨てようという話ではなく、順序を変えようという話です。究極の本物の問題は、今日の実力では解けません。学習だけが、明日もっと難しい問題を解けるようにしてくれるからこそ、HypoLoopは成果物よりも学習を優先します。小さなチャレンジは回り道ではなく、究極の問題へ上っていく唯一の階段なのです。

Core Concepts

四つの柱

4つの概念はすべて、同じ一つの例で解説します — 🍞 パンづくり。チームの目標は「地元で愛されるパン屋を開く」、私はまだ発酵を知らないメンバーです。

CONCEPT 01

仮説エンジン

情報がない状態でまず仮説を立て、検証可能な質問に分解し、探究しながら壊し、デルタを評価します。仮説が壊れる瞬間こそ最大の学習ポイントであり、検証の瞬間がそのまま試験になります。

🍞 パンで例えると

学ぶ前にまず書きます — 「パンはベーキングパウダーのおかげで膨らむと理解している。イーストを抜いて焼いてみれば分かるはずだ。」焼いてみると平らです。仮説が壊れるこの瞬間、酵母発酵という本当の原理が刻み込まれます。先に検索していたら、素通りしていた知識です。

CONCEPT 02

フラクタル入れ子

チームは世界の問題に向けた仮説を回し、個人はその中に自分の学習仮説を入れ子にします。「このチームの問題を解く過程で、私は意図的にXを学ぶ。」層は2つ、サイクルは1つです。

🍞 パンで例えると

チームは「うちの町の人たちは、朝に焼きたての健康パンを求めているはずだ」という仮説を回します。私はその中に「このパン屋を準備する過程で、私は発酵を学ぶ」という学習仮説を入れ子にします。チームはパン屋を開き、私は発酵を学ぶ — 層は2つ、サイクルは1つ。

CONCEPT 03

チャレンジ階層ツリー

究極の本物の問題を、挑戦可能なサイズまで階層的に分解します。どんなに小さなチャレンジも、本物の問題と血統でつながっています。おもちゃか足場かを決めるのは、サイズではなくつながりの可視性です。

🍞 パンで例えると

地元で愛されるパン屋を開く → 看板となる健康パンの開発 → サワードウを焼く → 天然酵母種をつくる(リーフ)。ガラス瓶で酵母種を育てる1週間の小さな挑戦も、ツリーをたどって上れば、パン屋という本物の問題と血統でつながっています。だからおもちゃではなく、足場なのです。

CONCEPT 04

学習デルタ評価

成否の基準は「問題が解けたか」ではなく「仮説と比べて何が変わったか」です。問題も本物、挑戦も本物ですが、採点されるのは変化量です。だからこそ、成果物の質の低下を公式に受け入れることができます。

🍞 パンで例えると

今日焼いたサワードウは、平らにつぶれて失敗しました。しかしサイクルの開始時点では、発酵温度が何を変えるのか知らなかった私が、今は「低温発酵が風味をつくり、過発酵が生地を崩す」と説明できます。採点されるのはパンの形ではなく、この距離(Δ)です。

Playbook

問題設定から振り返りまで、
八つのステップで運用します

運営者一人の力量ではなく、構造が回すフレームワークです。各ステップのテンプレートをコピーしてそのまま埋めれば、最初のサイクルが始まります。

STEP 1
問題設定

本物の問題を究極チャレンジとして掲げます

チームが本当に解決したい、世界に実在する問題を究極チャレンジとして宣言します。不可能ではないが簡単でもなく、メンバーの興味・関連性とつながっている必要があります。この問題の真正性が、ツリー全体の真正性になります。

私たちのチームは[世界の本物の問題P]を解決しようとしている。
この問題は[誰が、どんな状況で]実際に直面している問題だ。
解決されれば[何が変わったか]で確認できる。
進め方 — このとおりに進めてください
チーム全体60〜90分付箋・ホワイトボード
  1. 収集(15分) — 各自「直近6か月以内に自分が直接経験したか、すぐそばで目撃した不便」を付箋1枚に1つずつ、5個以上書きます。大げさなものである必要はありません。
  2. グルーピング(10分) — すべて壁に貼り、似たもの同士をまとめて、まとまりごとに名前を付けます。
  3. 実在検証(10分) — まとまりごとに一緒に問います:「これを実際に経験している人を、私たちは一人でも名指しできるか?」名指しできなければ除外します。
  4. 投票(5分) — 残った候補に1人2票:「最も興味深いもの」に1票 +「自分たちの規模で対応できるもの」に1票。
  5. 宣言(15分) — 最多得票の問題でテンプレートの3行を一緒に埋め、全員が声に出して読み、同意すれば確定します。
空欄を埋めるヒント質問
  • [世界の本物の問題P] — 誰かがすでに時間やお金を費やしながら耐えている不便か? 誰も惜しがらないなら、それは問題ではなくアイデアです。
  • [誰が、どんな状況で] — ペルソナではなく実在の人物レベルで。「20代の会社員」✗ →「通勤途中に朝食を取りそこねている隣のチームの田中さん」✓
  • [何が変わったか] — 解決された翌日に、目で観察できる場面一つで。
🍞 記入例「私たちのチームは『地元で焼きたての健康的なパンを買える場所がない』という問題を解決しようとしている。この問題は、朝食をパンで済ませる通勤途中の隣人たちが毎日直面している問題だ。解決されれば、朝7時に店の前に行列ができることで確認できる。」
STEP 2
ツリー分解

挑戦可能なサイズまで階層的に分解します

究極チャレンジを解決するために前提となる下位チャレンジを展開し、個人が1サイクルの中で挑戦できるサイズのリーフが出てくるまで分解します。各リーフから究極チャレンジまでの経路が常に見えるように、ツリーをチームの共有スペースに掲示します。

[上位チャレンジ]を解決するには
[下位チャレンジ A / B / C]が前提となる。

リーフ判定 — 一人が1〜2週間で仮説を
壊してみられるサイズか?(いいえ → もう一段分解)
ヒント: リーフ1つ = 1〜2週間で仮説を壊してみられるサイズ
進め方 — このとおりに進めてください
チーム全体60分大きな紙・Miro/FigJam
  1. ルート掲示 — 究極チャレンジを一番上に書きます。
  2. 分解質問の繰り返し — 「これが実現するには、先に何ができていなければならないか?」と問い、答えを2〜4個の下位チャレンジとして書きます。技術・ユーザー・運用の3方向から一度ずつ問えば、抜け漏れが起きにくくなります。
  3. リーフ判定 — 各ノードに問います:「一人が1〜2週間で仮説を壊してみられるサイズか?」いいえなら、そのノードで2を繰り返します。
  4. 掲示 — 完成したツリーをチームの共有スペース(Notion・壁)に貼ります。どのリーフからでもルートまでの経路が一目で見えなければなりません。
ヒント
  • ノードは名詞ではなく行動で — 「発酵の知識」✗ →「天然酵母種をつくってみる」✓
  • 深さは3〜4段で十分です。リーフが15個を超えたら、今回のサイクルで使う枝だけ残して畳んでおきましょう。
  • ツリーは完成品ではなく地図の草案です — サイクルごとに更新されます。
🍞 記入例パン屋を開く → 看板となる健康パンの開発 → サワードウを焼く → 天然酵母種をつくる(リーフ)
STEP 3
リーフ選択

各自が「できないこと」を選びます

白紙からチャレンジを立てるのではありません。ツリーのリーフの中から選びますが、ルールはただ一つ — 今できないことを選ぶ。初心者には基礎スキルが、熟練者には専門性を増幅する隣接領域(デザイン、ドメイン、市場)がリーフになります。

私は[リーフL]を選ぶ。
今の私にはこれができない — 根拠: [試みると起きること]。
このリーフは[リーフ → 中間 → 究極]の経路で
チームチャレンジとつながっている。
注意: 得意なことを選んだ瞬間、このフレームワークを使う理由が消えます
進め方 — このとおりに進めてください
個人20分完成したツリー
  1. スキャン — ツリーのリーフを眺めながら、「今やれと言われたらできないもの」に各自印を付けます。
  2. 選択 — 印を付けたものの中から、今回のチームサイクルで実際に使われるリーフを1つ選びます。
  3. 根拠の記述 — 「できないという根拠」を一行で書きます:今試みたら実際に起きること。
  4. 経路の朗読 — リーフから究極チャレンジまでの経路を、仲間に声に出して説明します。言葉に詰まったら、先にツリーを直します。
ヒント
  • 根拠が「やったことがないから」だけでは弱いです — 「やってみるとこの地点で詰まる」と書ければ良いリーフです。
  • 2つの間で迷ったら、より分からない方を選びます。
  • 専門家なら — 技術のリーフが全部簡単に見えるときがサインです。デザイン・ドメイン・市場の枝から選んでください。
🍞 記入例「私は『天然酵母種をつくる』を選ぶ。根拠: 小麦粉と水を置いておけば、ただカビが生えるだけだと思っている — 発酵と腐敗を区別できない。経路: 酵母種 → サワードウ → 看板パン → パン屋。」
STEP 4
仮説作成

学ぶ前に、今の理解で仮説を書きます

検索する前、講義を聞く前に、何の情報もなしに現在の理解だけで仮説テンプレートを埋めます。ぎこちなく、間違っているほど良いのです。この仮説が、以後に出会うすべての情報が引っかかる認知のフックになります。

私は今[対象X]を[このように理解/予想]している。
[検証行動Y]をやってみれば、この理解が正しいか分かるはずだ。
この仮説が壊れたら、私は[分かるはずのこと]を知ることになる。
進め方 — このとおりに進めてください
個人15分検索禁止
  1. 遮断 — タイマー15分。検索窓・講義・AIをすべて閉じます。
  2. 作成 — いま頭の中にある理解だけで、テンプレートの3行を埋めます。
  3. そのままにする — ぎこちなくて間違っている気がしても、直しません。この文書の目的は正解ではなく壊れるための標的です。
  4. サイズ確認 — 検証行動Yが1〜2週間で実行できるサイズかどうかだけを確認します。
空欄を埋めるヒント質問
  • [対象X]が思い浮かばなければ — リーフを始めたとき最初にぶつかるものがXです。
  • [検証行動Y]は机の外の行動で — 読む・視聴する ✗、つくる・測る・自分で経験する ✓
  • [分かるはずのこと]が書けなければ、仮説がまだ予測になっていないということ — Xをもっと絞りましょう。
🍞 記入例「私は今、パンはベーキングパウダーのおかげで膨らむと理解している。イーストを抜いて焼いてみれば、この理解が正しいか分かるはずだ。この仮説が壊れたら、私は生地を膨らませる本当の原理を知ることになる。」
STEP 5
ピア検証

仲間が一つの質問を投げかけます

仮説を立てた本人ではなく、チームの仲間が尋ねます — 「この仮説が壊れたら、君は何を知ることになる?」答えられなければ、仮説を磨き直して書き直します。このリチュアルによってループキーパーなしでも検証が回り、尋ねる側もメタ認知のトレーニングになります。ループキーパーは介入者ではなく、この構造がうまく回っているかを見守るインフラ管理者です。

質問: 「この仮説が壊れたら、君は何を知ることになる?」
合格 — 答えが具体的な理解を一つ指している。
差し戻し — 検証行動が曖昧だ / 知ることになるものがない
       / すでに知っていることだ。
差し戻されたら、仮説を磨いて書き直す。
進め方 — このとおりに進めてください
2人1組1人あたり10分
  1. 朗読 — 仮説を声に出して読みます。
  2. 質問 — 仲間が尋ねます:「この仮説が壊れたら、君は何を知ることになる?」
  3. 判定 — 30秒以内に、具体的な理解一つで答えられれば合格。
  4. 差し戻し基準 — 検証行動が曖昧だ / 知ることになるものがない / すでに知っていることだ。
  5. 再挑戦 — 差し戻されたら、その場で直してもう一度尋ねます。往復3分を超えないようにします。
ヒント
  • 尋ねる側はアドバイス禁止 — 質問だけをします。直すのは本人の仕事です。
  • 最高の差し戻しは「それ、君はもう知っているよね。」
  • ループキーパーがいる場合は、このリチュアルが回っているかだけを見ます — 問答には割り込みません。
🍞 問答例Q「壊れたら何を知ることになる?」— A「膨らみの原因が化学膨張剤なのか、生物学的な発酵なのかを区別できるようになる。」→ 合格
STEP 6
探究

チームの問題を解きながら仮説を壊します

別枠の勉強時間ではなく、チームの問題を解くまさにその過程が仮説の実験場です。仮説が壊れる瞬間を、その場で記録してください — 予想と現実のギャップが大きいほど、更新される理解も大きくなります。

[探究ログ — 仮説が壊れた瞬間ごとに]
予想: [予想していたこと]
現実: [実際に起きたこと]
ギャップから学んだこと: [更新された理解]
仮説が壊れる瞬間が最大の学習ポイント — 後回しにせず、その場で記録
進め方 — このとおりに進めてください
個人サイクルの間ずっと探究ログ
  1. いつもどおり — 別枠の勉強時間はつくりません。チームの作業をそのまま進めます。
  2. 即時記録 — 「あれ、予想と違う?」と思った瞬間ごとに、ログ3行(予想/現実/学んだこと)をその場で5分以内に書きます。
  3. 週次共有 — 週1回のスタンドアップで、各自壊れた仮説を1つ共有します。自慢する場です。
ヒント
  • ログは上手に書く必要はありません — 残しさえすればいいのです。写真・ボイスメモもログです。
  • 1週間経っても何も壊れないなら — 検証行動をもっと強く、あるいは仮説をもっと危険に。
🍞 ログ例予想: イーストがなければまったく膨らまない / 現実: 1日置いたら少し膨らんでいた / 学んだこと: 空気中にも野生酵母がいる。
STEP 7
デルタ振り返り

成果物ではなく変化量を振り返ります

サイクルの終わりに、各自が開始時点の仮説をもう一度取り出し、当時の理解と今の理解の距離(デルタ)を測ります。そして、その学びがチームチャレンジをどこまで前進させたかも一緒に記録します — 学習と解決はここで一つに出会います。

始まりの私は[このように理解]していた。
今の私は[このように理解]している。
デルタ(Δ): [何が更新されたか]
解決への貢献: [この学びがチームチャレンジを
どこまで前進させたか]
採点されるのは成果物ではなくデルタ — 結果が拙くても、デルタが大きければ成功です
進め方 — このとおりに進めてください
チーム全体60分STEP 4の仮説の原文
  1. 原文朗読 — 各自、STEP 4の仮説を修正せずそのまま読み直します。
  2. デルタ作成 — 「サイクル開始時の自分」に説明するつもりで、デルタのテンプレートを埋めます。
  3. 発表 — 順番に1人3分。成果物のデモではなく、理解の変化だけを話します。
  4. チーム層を一文で — 今回のサイクルがチーム仮説をどこまで前進(または棄却)させたかを合意します。
ヒント
  • デルタがつかめなければ、STEP 6のログをもう一度開いてください — 壊れた瞬間ごとにデルタが1つずつあります。
  • 「知らなかった → 知っている」よりも、「こう理解していた → ああ理解している」の方が良いデルタです。
  • 振り返りの冒頭で、この一文を声に出して読んでください — 「結果が拙くても、デルタが大きければ成功だ。」
🍞 記入例「始まりの私は、膨らみは膨張剤のおかげだと理解していた。今の私は、酵母が糖を分解しながら出すCO₂が原理であり、膨張剤はその近道だと知っている。Δ: 化学 → 生物。貢献: 看板サワードウ試作品1号。」
STEP 8
次のサイクル

更新された理解で次の仮説を立てます

振り返りで更新された理解が、次のサイクルの出発点です。次のリーフはもっと分からないものを選び、チーム層も同じリズムで回ります — チーム仮説の棄却は失敗ではなくピボットの根拠です。ループは止まりません。

次のリーフ: [今回よりもっと分からないもの]
次の仮説: [更新された理解で書き直す]
チーム層への反映: [棄却されたチーム仮説があれば → ピボットの方向]

ループキーパーは教えません — 質問します

ループキーパー(ファシリテーター)は、知識を伝える人でも、チャレンジを検査する人でもありません。構造がうまく回っているかを見守るインフラ管理者であり、必要な瞬間に質問を一つ投げかける人です — ループが止まらないように守る人、だからキーパーです。答えは与えません — 答えを探す過程こそが学習者のサイクルだからです。以下の質問はループキーパーの専有物ではありません。キーパーがいなくても、チームメンバー同士で投げかけ合うだけでフレームワークは回ります。

リーフを選ぶとき
  • それ、今でもできるんじゃない? できないという根拠は何?
  • そのリーフから究極チャレンジまでの道のりを、言葉で説明してくれる?
仮説を立てるとき
  • この仮説が壊れたら、君は何を知ることになる?
  • 検索したり学んだりする前に書いた仮説だよね?
  • その検証行動、1〜2週間で実際にやってみられる?
探究の途中で
  • 最近、予想と違う展開になった瞬間はあった? どこに記録した?
  • 仮説がまだ一つも壊れていないなら — 安全に立てすぎたんじゃない?
サイクルを閉じるとき
  • サイクル開始時の君なら、これを今どう説明していたと思う?
  • 今回学んだことは、チームチャレンジをどこまで前進させた?
  • 更新された理解で立てる、次の仮説は何?

SAMPLE 01🍞 地元のパン屋サイドプロジェクト4人 · 全員が非専門の初心者
究極チャレンジ
私たちのチームは『地元で焼きたての健康的なパンを買える場所がない』という問題を解決しようとしている。朝食をパンで済ませる通勤途中の隣人たちが毎日直面している問題だ。解決されれば、朝7時に店の前に行列ができることで確認できる。
ツリー → リーフ
パン屋を開く → 看板となる健康パンの開発 → サワードウを焼く → 天然酵母種をつくる
選択の根拠
「小麦粉と水を1週間置いたらカビが生えると思っている — 発酵と腐敗を区別できない。」
仮説
私は今、パンはベーキングパウダーのおかげで膨らむと理解している。イーストを抜いて焼いてみれば分かるはずだ。この仮説が壊れたら、私は生地を膨らませる本当の原理を知ることになる。
ピア検証
Q「壊れたら何を知ることになる?」— A「膨らみが化学膨張剤のせいなのか、生物学的な発酵のせいなのか。」→ 合格
壊れた瞬間
予想: イーストを抜けばまったく膨らまない / 現実: 1日置いたら『少し』膨らんでいた / 学んだこと: 空気中に野生酵母が生きている — 発酵は加えるものではなく、起きるものだ。
Δ デルタ:「膨らみ = 膨張剤の化学反応」→「膨らみ = 酵母が糖を分解しながら出すCO₂。膨張剤はその近道。」チームへの貢献: 看板サワードウ試作品1号。
次のサイクル
低温発酵と風味の関係 — 「発酵は長いほど良い」という次の仮説へ。
SAMPLE 02🦯 屋内ナビゲーション教育プログラムのチーム5人 · iOS専攻の学生
究極チャレンジ
私たちのチームは『視覚障害者が屋内で道に迷う』という問題を解決しようとしている。盲導犬も点字ブロックも届かない建物の内部で毎日起きている問題だ。解決されれば、初めての建物で目的地まで一人でたどり着けることで確認できる。
ツリー → リーフ
屋内ナビゲーション → 目を使わずに伝える(非視覚インターフェース)→ VoiceOverのジェスチャー体系を、目を閉じて一日使ってみる
選択の根拠
「3年間iOS開発を学んでいるのに、VoiceOverを一度もオンにしたことがない。」
仮説
私は今、視覚障害者は画面を上から下へ順番に読み上げながら使うと理解している。目を閉じて一日使ってみれば分かるはずだ。この仮説が壊れたら、私は非視覚ナビゲーションの実際の戦略を知ることになる。
ピア検証
Q「壊れたら何を知ることになる?」— A「線形の読み上げなのか、それとも別の探索戦略があるのか。」→ 合格
壊れた瞬間
予想: 順番に聞きながら使う / 現実: 半日で断念 — 遅すぎて誰もそうは使わない。ローターと見出しジャンプで『構造を飛び越えながら』探索する / 学んだこと: 非視覚での使用の核心は読み上げではなく構造だ。
Δ デルタ:「線形読み上げモデル」→「構造ジャンプモデル。良いアクセシビリティ = よく設計された見出しの階層。」チームへの貢献: 案内UXを「順番に読む」から「ランドマーク + 方向」方式にピボット。

SAMPLE 03📦 市場の在庫アプリブートキャンプのチーム6人 · フロントエンド一筋の受講生
究極チャレンジ
私たちのチームは『昔ながらの市場の商人が、帳簿なしでは在庫を把握できない』という問題を解決しようとしている。閉店後の手書きの棚卸しに毎日1時間を費やす商人たちの問題だ。解決されれば、店じまいの整理が10分以内に終わることで確認できる。
ツリー → リーフ
在庫アプリ → 在庫データが安全に蓄積される → 変動記録のAPIサーバーを立てる
選択の根拠
「サーバーはいつも他人がつくったものをfetchするだけだった。ポートが何かを説明できない。」
仮説
私は今、サーバーの保存はリクエストが来るたびにファイルに書き込めば十分だと理解している。リクエストを100件同時に送ってみれば分かるはずだ。この仮説が壊れたら、私は同時実行で何が問題になるのかを知ることになる。
ピア検証
Q「壊れたら何を知ることになる?」— A「ファイルへの書き込みが、同時リクエストでどうなるのか。」→ 合格
壊れた瞬間
予想: リクエストが多ければ少し遅くなるだけ / 現実: 同時100件の後、在庫数が実際と食い違った — エラーも出さずに静かに狂う(race condition)/ 学んだこと: 保存は『書き込み』ではなく『競合する書き込み』だ。
Δ デルタ:「保存 = ファイルに書く」→「保存 = 競合する書き込み。トランザクションとロックがなぜ存在するのかを体で知っている。」チームへの貢献: SQLiteのトランザクションを導入し、市場の商人1人とベータを開始。

地下店舗の途切れがちなネットワーク — オフラインファーストの同期へ。
SAMPLE 04🔎 社内Wiki検索ボット1人(一人で使うHypoLoop) · バックエンド3年目

私は『新人が同じ質問をSlackで繰り返す』という問題を解決しようとしている。会社のWikiに答えがあるのに見つけられない問題だ。解決されれば、オンボーディングの質問の半分がボットで完結することで確認できる。一人で使う場合: チーム層 = 自分がつくりたい本物の成果物。
ツリー → リーフ
検索ボット → 質問と文書を意味でつなぐ → 文書をベクトルに変換して類似度検索を試してみる
選択の根拠
「『埋め込み(エンベディング)』を比喩でさえ説明できない。」
仮説
私は今、埋め込み検索はキーワード検索の上位互換だと理解している。同じ質問20個を両方式に投げて比較すれば分かるはずだ。この仮説が壊れたら、私は2つの方式がそれぞれいつ勝つのかを知ることになる。
ピア検証
仲間がいないため、ブログに仮説を先に公開した — 未来の自分と読者が検証者。(一人で使うときの代替リチュアル)
壊れた瞬間
予想: 埋め込みが全領域で優勢 / 現実: 固有名詞・誤字の質問ではキーワード検索が圧勝 / 学んだこと: 埋め込みは『似た意味』に強く、『正確にその単語』に弱い。
Δ デルタ:「上位互換」→「相互補完 — だからハイブリッド検索が存在する。」成果物: キーワード+ベクトルのハイブリッド検索プロトタイプ。

SAMPLE 05🎛 デプロイダッシュボード社内ハッカソン3人 · 10年目のバックエンド開発者
究極チャレンジ
私たちのチームは『障害のたびに、今何がデプロイされているのかを互いに聞き合う』という問題を解決しようとしている。デプロイの状況が開発者の頭の中にしかない問題だ。解決されれば、非開発職でも状況を30秒以内に自分で確認できることで検証できる。

ダッシュボード → 一目で読める画面をつくる → 情報設計をデザイナーなしで自分でやる

Δ デルタ:「良い画面 = 漏れなく」→「良い画面 = 一つの画面が一つの質問に答える。」チームへの貢献: 30秒テストを通過したv2でハッカソン2位。

FAQ

よくある質問

成果物の質が下がっても、本当に大丈夫ですか?
大丈夫どころか、設計の一部です。HypoLoopで採点されるのは成果物ではなく学習デルタだからです。ただし、問題そのものは本物でなければならず、挑戦も本物でなければなりません — 真正性は階層ツリーが守ってくれます。すべての小さなチャレンジが究極の本物の問題と血統でつながっている限り、拙い成果物も足場であって、おもちゃではありません。
熟練した専門家は、このフレームワークで何を学ぶのですか?
ルールは同じです — できないことを選ぶ。ただし専門家の「できないこと」は基礎スキルではなく、自分の専門性を増幅する隣接領域です。10年目の開発者なら、より良い成果物のために必要なデザイン感覚や市場のドメイン知識がリーフになります。得意なことをさらに得意にするための周辺知識の習得が、専門家のチャレンジです。
ループキーパー(ファシリテーター)がいなくても運用できますか?
それこそが目標です。ツリー分解が関連性と難易度を、仮説テンプレートが検証可能性を、ピア検証のリチュアルが品質を担います。ループキーパーがいる場合、その役割は介入ではなくこの構造物がうまく回っているかを見守るインフラ管理です。勉強会や小規模チームなら、キーパーなしのピア検証だけで始められます。
チームの成果物と個人の学習が衝突したら、どうしますか?
個人の学習が優先します。これは例外規定ではなく、フレームワークのアイデンティティです — HypoLoopは問題解決のフレームワークではなく学習のフレームワークであり、学びは最終的に個人に起きなければならないからです。この原則のおかげで、不慣れなパートを引き受ける行動が迷惑ではなく正しい使い方になります。
一人でも使えますか?
使えます。チーム層を「自分がつくりたい本物の成果物」と置き、それを階層ツリーに分解した上で、リーフごとに仮説サイクルを回せばよいのです。ピア検証は、勉強仲間やコミュニティ、あるいは未来の自分に残す記録で代替できます。核心 — 学ぶ前にまず仮説を書く — は、一人のときも同じように機能します。
既存のチャレンジ型メソッドと併用できますか?
HypoLoopは特定メソッドの改良版ではない独立したフレームワークですが、チャレンジがもたらす動機のエネルギーを継承しているため、チャレンジ型カリキュラムの上に個人の学習層を重ねる形で組み合わせられます。既存のチームチャレンジを究極チャレンジと置き、ツリー分解と個人の仮説サイクルを追加するところから始めてみてください。
